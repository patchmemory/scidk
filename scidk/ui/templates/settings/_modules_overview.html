Modules Overview

Reference guide for SciDK's module types, contracts, and validation lifecycle.

SciDK uses a modular architecture where different component types handle specific tasks. Each module type has a well-defined contract specifying what inputs it expects, what it writes to the graph, and how validation applies. This page serves as a technical reference for developers, collaborators, and IDE agents building conformant modules.

Module Types

Module Type Purpose Inputs Writes to Graph Authoring Surface
Interpreters Transform raw files into structured graph data File path, file type File nodes, metadata properties, extracted entities Scripts page, Files page
Plugins Extend SciDK with custom endpoints and integrations HTTP requests, plugin config Custom labels, relationships, metadata (plugin-defined) Plugins page
Links Create relationships between nodes via matching rules Node properties, matching config, external data (optional) Relationships with confidence scores and provenance Links page
Labels Define custom node types with schema constraints Label definition, property schema Label constraints, index definitions, validation rules Labels page
Analyses Run analytical computations over graph data Cypher query or Python script, parameters Result nodes, computed properties, analysis metadata Scripts page, Results page
Scripts General-purpose graph operations and automation User-defined parameters, graph context Any graph modifications (user-defined) Scripts page

Validation Lifecycle

Most module types follow a three-state validation lifecycle to ensure safe deployment:

Draft

Newly created or edited modules. Not executed automatically. Available for manual testing via authoring interface.

Validated

Passed validation tests. Safe for use. Validation checks syntax, runtime dependencies, and contract conformance.

Active

Validated and enabled for automatic execution. Applies to system workflows (interpreters, links, plugins). Scripts/analyses remain validated but run on-demand.

Note: Plugins and Labels may have simplified or alternate validation flows depending on their registration mechanism. Check the specific module settings page for details.

Configuration

Module-type-level configuration (not per-module settings) is managed in the Settings panel:

  • Modules > Interpreters — Table format registry, default file extension assignments
  • Modules > Plugins — Active plugin health, registered API endpoints (read-only)
  • Modules > Links — Global fuzzy matching algorithm and threshold defaults
  • Modules > Labels — Schema-level settings (if applicable)
  • Modules > Analyses — Analysis health monitoring
  • Modules > Scripts — Script runtime health monitoring
